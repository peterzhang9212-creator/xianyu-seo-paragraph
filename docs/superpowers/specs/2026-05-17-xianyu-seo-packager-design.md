# 闲鱼教师 SEO 封装程序设计文档

## 1. 背景与目标

当前闲鱼自动发帖流程依赖多个分散脚本：

- 商品素材文件夹组装
- 图片批量加水印
- 子文件夹名导出
- 上传表格生成
- BitBrowser 驱动的闲鱼自动发布

这些脚本分散在不同目录，彼此通过文件产物串联，日常使用和后续维护成本都偏高。一旦原目录被移动或删除，整套流程就会失效。

本项目的目标是在 `D:\2 搞钱项目\程序测试\好用的封装程序\闲鱼教师SEO项目` 内构建一个可独立运行的封装程序，满足以下要求：

- 所有核心功能都收口到本项目目录内，删除原脚本目录后仍可运行
- 用 4 步交互界面引导执行：`文件组装 -> 加水印 -> 生成上传表 -> 发布设置`
- 每一步都能记录上一次输入和默认值
- 步骤之间不强绑定，可以跳过中间步骤直接操作后续步骤
- 若存在明确的上一步输出结果，可安全自动带入到下一步
- 生成上传表时内置正文模板导入能力，不再要求单独准备“子文件夹名导出表”
- 水印步骤固定为“主图和详情图都加”，并且固定使用独立输出模式，保留原图
- 提供使用说明和维护说明，方便后续快速调整规则

## 2. 非目标

第一版不包含以下内容：

- 不重新设计闲鱼发布底层自动化策略，只在现有可用逻辑基础上做迁移和封装
- 不做云端同步、多人协作、权限系统
- 不强制把所有路径流程做成项目工程概念；第一版仍允许用户按步骤自由选路径
- 不在第一版中加入正文内容的可视化逐条编辑器，正文来源以导入模板文件为主

## 3. 总体方案

推荐方案为：`GUI 外壳 + 本地核心模块 + 独立项目目录`

设计原则：

- GUI 负责顺序引导、参数输入、路径选择、日志显示、状态回填
- 核心逻辑按功能拆分为独立模块，避免把所有逻辑塞进单文件脚本
- 现有分散脚本中的有效逻辑迁移到本项目目录中，不再在运行时依赖旧目录
- 统一通过配置文件保存“上次输入值”和默认参数
- 每一步独立校验自己的输入条件，不要求前一步必须已经执行

相比“GUI 直接调用旧脚本”的方案，这一方案更符合“旧目录删除后仍可用”的要求，也更利于后续维护。

### 4.1 目录职责

- `app.py`
  程序启动入口，负责启动 GUI。
- `config/defaults.json`
  存储最近使用的路径、默认参数、上一步输出记录。
- `core/`
  放真正的业务逻辑。
- `ui/`
  放界面层代码。
- `vendor/xianyu_uploader/`
  放从现有闲鱼自动上传项目迁移过来的稳定发布核心。
- `templates/content_template.xlsx`
  内置正文导入模板。
- `workspace/`
  放程序默认生成的中间结果和输出文件。
- `logs/`
  放运行日志。
- `docs/`
  放面向使用和维护的说明文档。

## 5. 四步流程设计

### 5.1 第一步：文件组装

#### 输入

- 城市/序号 Excel
- 父目录
- 固定后缀文本
- 封面图目录
- 详情图目录

#### 处理逻辑

- 读取 Excel 中的 `序号` 和 `城市名`
- 生成子文件夹，命名规则为 `城市名 + 固定后缀`
- 按序号匹配封面图
- 把封面图复制并统一命名为 `商品主图.jpg`
- 把详情图复制到每个子文件夹
- 在每个子文件夹内生成 `商品标题.txt`

#### 输出

- 一批规范化商品素材文件夹
- 组装日志
- 组装结果统计

#### 固定约束

- `商品主图.jpg` 是发布步骤依赖的固定主图文件名
- `商品标题.txt` 默认写入文件夹名，作为辅助文本文件保留

### 5.2 第二步：加水印

#### 输入

- 商品素材根目录
- 水印文字
- 字号倍率
- 间距系数
- 透明度

#### 固定规则

- 主图和详情图都加水印
- 只使用独立输出模式，不覆盖原图

#### 处理逻辑

- 扫描商品素材目录中的图片
- 对每个商品文件夹内的主图和详情图都执行加水印
- 保留目录结构，将结果输出到独立目录
- 记录超体积或处理失败的文件

#### 输出

- 水印输出根目录
- 水印处理日志
- 超体积提示清单

### 5.3 第三步：生成上传表

#### 输入

- 用户每次手动选择的正文内容文件
- 商品素材根目录
- 可选的手动指定发布素材根目录

#### 规则来源

上传表模板和列结构沿用现有 `generate_upload_sheet.py` 的生成逻辑，内置固定模板，不要求用户手工再维护“子文件夹名导出表”。

#### 处理逻辑

- 读取正文内容文件
- 自动扫描商品素材根目录下的一层子文件夹名
- 用正文中的商品名与子文件夹名做一一匹配
- 使用既定模板输出上传 Excel

#### 匹配规则

- 正文文件必须包含表示商品名的列
- 该列值必须与素材子文件夹名完全一致
- 行数不一致或名称不一致时，直接报错并列出差异，不做静默修正

#### 输出

- 标准上传 Excel
- 校验报告或界面错误提示
- 可选素材清单预览

#### 关键变化

原“获取子文件夹名”能力不再作为独立步骤存在，而是并入本步骤内部自动完成。

### 5.4 第四步：发布设置

#### 输入

- 上传 Excel
- 规格 1 名称和价格
- 规格 2 名称和价格
- 库存
- 起始网址

#### 运行依赖

- 本机已安装 Python 依赖
- BitBrowser 已打开并可访问闲鱼页面
- 上传 Excel 格式符合系统要求

#### 处理逻辑

- 读取 Excel 中 `发布情况` 为空的记录
- 从素材目录读取 `商品主图.jpg` 和详情图
- 调用迁移后的闲鱼自动发布核心
- 成功、失败或待确认结果写回 Excel
- 输出运行日志和结果统计

#### 输出

- 回写状态后的上传 Excel
- 发布日志
- 发布结果统计

## 6. 步骤解耦与自动带入

### 6.1 解耦原则

步骤之间不是强依赖关系。用户可以：

- 跳过水印，直接用组装素材生成上传表
- 直接选择现成素材目录来生成上传表
- 直接选择已有上传 Excel 进入发布步骤

系统不强制按 1 到 4 顺序执行，但每一步在执行前必须独立通过自身校验。

### 6.2 自动带入规则

允许自动带入，但必须遵循“安全优先”原则。

#### 可自动带入

- 第一步成功后，记录“最近组装输出目录”
- 第二步默认读取“最近组装输出目录”
- 第二步成功后，记录“最近水印输出目录”
- 第三步优先默认带入“最近水印输出目录”，若没有则回退到“最近组装输出目录”
- 第四步默认带入“最近生成的上传 Excel”

#### 可自动识别但需确认

- 检测到目录中存在典型水印输出目录时，提示用户是否采用
- 检测到多个候选目录时，列出候选项而不是直接静默选择

#### 不自动兜底

- 正文文件与素材目录数量不一致
- 商品名与子文件夹名不一致
- 发布所需 Excel 与素材目录关系不明确

## 7. 默认值和状态保存

程序需持久化记录以下内容：

- 每一步最近一次输入的路径
- 固定后缀
- 水印文字、字号倍率、间距系数、透明度
- 最近一次正文文件路径
- 最近一次生成的上传表路径
- 最近一次组装输出目录
- 最近一次水印输出目录
- 发布默认参数：规格、价格、库存、起始网址

建议统一保存在 `config/defaults.json` 中，由独立模块负责读写和兼容默认值。

## 8. 关键数据边界

为了保证多步骤串联稳定，系统内部统一使用以下关键标识：

- `商品文件夹名`
  作为素材、正文匹配、发布目录定位的核心主键
- `商品主图.jpg`
  作为发布主图固定文件名
- `商品标题.txt`
  作为素材目录内辅助文本
- `发布情况`
  作为发布结果回写列

## 9. 模块职责设计

### 9.1 `core/assembly.py`

- 迁移并整理 `folder_batch_processor_V2.py` 中的可复用逻辑
- 提供非交互函数接口，供 GUI 直接调用
- 负责路径清洗、Excel 解析、文件夹创建、主图与详情图复制、标题文件生成

### 9.2 `core/watermark.py`

- 迁移 `batch_watermark_agent.py` 的核心逻辑
- 提供目录级批量加水印函数
- 固定支持主图和详情图统一加水印
- 固定使用独立输出模式

### 9.3 `core/content_import.py`

- 负责读取正文模板文件
- 校验必填列
- 返回规范化内容记录

### 9.4 `core/upload_sheet.py`

- 迁移并整理 `generate_upload_sheet.py` 的模板生成逻辑
- 直接扫描素材目录，不再依赖单独的“子文件夹名导出脚本”
- 负责名称匹配、上传 Excel 生成和错误提示

### 9.5 `core/publish_runner.py`

- 作为发布步骤的统一入口
- 与 `vendor/xianyu_uploader/` 中的底层发布逻辑对接
- 控制 Excel 读取、状态回写、批量执行和运行摘要

### 9.6 `core/project_state.py`

- 读写 `config/defaults.json`
- 管理“最近使用路径”和“最近一步输出”
- 向界面层提供默认值和自动带入结果

### 9.7 `core/models.py`

- 统一放跨步骤的数据结构和结果对象
- 明确每一步输入输出的数据边界

## 10. 界面设计

### 10.1 总体布局

- 顶部：4 步导航条
- 中部：当前步骤操作区
- 底部或右侧：当前项目状态卡片
- 日志区：显示当前步骤执行信息

### 10.2 每一步通用控件

- 路径输入框
- `选择...` 按钮
- `自动识别` 按钮
- `开始执行` 按钮
- 结果摘要区
- 运行日志区

### 10.3 状态卡片内容

- 当前项目根目录
- 最近组装输出目录
- 最近水印输出目录
- 最近正文文件路径
- 最近上传表路径
- 最近运行时间

## 11. 错误处理策略

### 11.1 界面层

- 用中文提示当前步骤可理解的错误原因
- 不显示难懂的底层 traceback 给普通用户

### 11.2 日志层

- 保留完整技术错误详情
- 每一步生成独立日志条目
- 支持后续维护定位

### 11.3 分步骤校验

#### 文件组装

- Excel 是否存在
- Excel 必填列是否存在
- 封面图目录和详情图目录是否存在
- 序号是否能匹配封面图

#### 加水印

- 素材目录是否存在
- 是否扫描到可处理图片
- 输出目录是否可写

#### 生成上传表

- 正文文件是否存在
- 正文模板列是否完整
- 正文条数和素材子文件夹数是否一致
- 正文商品名和子文件夹名是否一致

#### 发布设置

- 上传 Excel 是否存在
- Excel 列结构是否完整
- 素材目录是否存在
- `商品主图.jpg` 是否存在
- BitBrowser 是否已打开

## 13. 验收标准

第一版至少满足以下验收项：

- 可以在新项目目录内独立启动 GUI
- 第一步可独立产出规范素材目录
- 第二步可对主图和详情图输出独立水印结果，且不覆盖原图
- 第三步可用正文模板文件和素材目录直接生成上传表
- 第四步可读取上传表并进入发布流程
- 程序重启后能恢复最近输入值和默认参数
- 删除旧的分散脚本目录后，新项目中的前 1 到 3 步仍可运行
- 第四步在满足本机环境依赖时可运行；真实发布结果仍受 BitBrowser 登录状态和闲鱼页面状态影响

## 14. 风险与后续关注点

### 14.1 页面变化风险

闲鱼发布页面可能变更，第四步底层自动化逻辑未来仍可能需要微调选择器或交互方式。

### 14.2 模板变更风险

若正文模板列名、上传表模板、发布 Excel 列头将来变化，需要优先调整：

- `core/content_import.py`
- `core/upload_sheet.py`
- `vendor/xianyu_uploader/` 中的 Excel 解析逻辑

### 14.3 文件命名强约束

系统依赖 `商品主图.jpg` 与子文件夹名一一对应。若未来命名规则变化，必须同步调整上传与发布逻辑。

## 15. 实施建议

实现阶段建议按以下顺序推进：

1. 先搭建项目目录和配置/文档骨架
2. 迁移文件组装和水印核心逻辑
3. 完成正文导入与上传表生成
4. 迁移并接通发布核心
5. 最后接 GUI 和状态保存

这样可以先确保核心能力独立可用，再把界面层接上，减少排查成本。
